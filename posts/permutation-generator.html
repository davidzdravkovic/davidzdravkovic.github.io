Designing a Permutation Generator From Its Invariant

Published {YYYY-MM-DD} · 8 min read

This post describes how I built a permutation generator by first understanding what any correct solution must satisfy, then designing around that constraint. No framework, no reference implementation — derived from the problem itself.

The problem

Input: a vector of n distinct elements. Output: every distinct ordering, exactly once. Total outputs: n!.

The first instinct is to try a swap pattern — move elements around and collect results. The problem is guaranteeing no duplicates and no gaps without knowing the correct pattern upfront. Random swapping requires tracking every previous result, which grows with n!. There has to be a structural guarantee instead.

The invariant

Before writing any code, ask: what must be true of every correct generator?

All n positions and all n elements are symmetric — no element is special, no position is special. By symmetry, each element must occupy each position the same number of times across all n! permutations:

n! / n = (n - 1)!
Each element occupies each position exactly (n − 1)! times across all n! permutations.

This is the constraint the design must satisfy. It also reveals a recursive structure: if one element is fixed at a position for (n − 1)! permutations, the remaining n − 1 elements must produce all their own permutations around it — the same problem, one size smaller.

The engine: three positions

The smallest useful base is n = 3. With a fixed swap rule — the anchor swaps with the element to its right; when it reaches the end, it swaps back with the first — six steps produce all 3! = 6 orderings of any three elements. Each element appears at each position exactly twice, which is (3 − 1)! = 2. Verifiable by enumeration.

This is the engine. Its contract: given any three distinct elements, emit all six orderings with each element at each position exactly twice.

The scheduler

For n > 3, the remaining elements sit in a prefix outside the engine. After the engine completes one full cycle of six permutations, the scheduler swaps a new element from the prefix into the engine. A bookkeeping map per depth records which elements have already occupied that depth's fixed position — each element is allowed there exactly once. When all elements have been through a depth, the depth above advances and the depths below reset.

This is a mixed-radix counter. The total work decomposes as:

n! = 3! × 4 × 5 × ... × n

The engine is the base digit. Each outer depth multiplies by the number of available elements at that depth. The bookkeeping enforces the (n − 1)! invariant at every level.

Note on originality

The algorithm belongs to a known family — factorial number system enumerations, of which Heap's algorithm is the most cited. Heap's bottoms out at n = 2; this one bottoms out at n = 3 with a hand-designed engine. I did not know this when I wrote it. The reason to share it is not novelty — it is that the invariant was the starting point, not the code.
